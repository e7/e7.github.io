# Redis

# 缓存一致性

缓存的数据一定要带TTL作为最终一致的兜底策略。

  • cache aside模式

    写操作

    1. 更新数据库。
    2. 删除缓存，不使用update的原因是并发更新的时候不能保证更新的是最新的数据。
    3. 更保险可以加上cdc（比如canal (opens new window)）覆盖缓存（使用set）。

    读操作

    1. 读数据库。
    2. 回填缓存，一定要用setnx，因为读取到的数据小概率是脏数据的。

  • CRDT

    • Conflict-free Replicated Data Type (opens new window)

# Tips

  • 缓存穿透，核心就是只让一个人取db

    • Singlefly，对关键字进行一致性哈希，使其某一维度的key一定命中某个节点，然后节点内使用互斥锁保证归并回源，但不能处理批量查询的情况。
    • Lease，就是限流，秒杀也可以使用类似机制，只有拿到token的请求才能访问数据库。
    • 空缓存保护（请求在cache和db都查不到）
      • 通过一些算法让请求参数不可枚举，直接在算法层面挡掉大部分恶意请求。
      • 查不到的数据，写个TTL较短的空缓存

  • 热点缓存

    • 小表广播，热点数据本地cache，App定时更新，甚至运营平台支持广播刷新本地cache
    • 对key加后缀，#1、#2之类，一定会miss，然后就会从db捞然后缓存
    • 热点数据发现

  • 每个微服务独享redis（或redis cluster）

  • 有界负载一致性哈希 (opens new window)

  • 拆分key，同一个hashset会落到同一个节点上，hashset过大会导致内存及请求的不均匀。

  • 序列化使用protobuf，尽可能降低内存占用。

  • 增量更新一致性：先EXPIRE再ZADD/HSET等操作，保证索引结构体务必存在的情况下去操作新增数据。

  • Bitset，存储每日登录用户，为了避免BITSET过大或者热点，需要使用region sharding，比如按照mid求余%和/1000，商为key，余数作为offset。

  • List：当堆栈使用

  • Sortedset：翻页、排序、有序的集合，杜绝zrange或zrevrange返回的集合过大。

  • Hashset：杜绝hgetall。

  • String：使用SET的EX/NX等扩展指令，SETNX可用于分布式锁，SETEX则是原子化的SET+EXPIRE

  • 避免超大value。
