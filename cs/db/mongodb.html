# 索引基础

注意权限

只有属于dbAdmin和dbAdminAnyDatabase角色的用户才能创建或删除索引

  • 可以使用createIndex()函数来为其它（除_id之外）字段创建索引，创建索引时需要指定排序规则（1升序，-1降序）。

    db.COLLECTION_NAME.createIndex({创建索引的键: 排序规则},{创建索引的参数（可选参数）})

    索引参数说明

    参数 数据类型 默认值 功能
    background Boolean false 是否后台创建索引（创建索引时不阻塞其它数据库操作）
    unique Boolean false 是否创建唯一索引
    name String 指定索引名称，如未指定，则会自动生成
    partialFilterExpression document 指定过滤表达式（不为所有记录创建索引，而只为满足条件的记录创建）
    sparse Boolean false 稀疏索引，对文档中不存在的字段不启用索引
    exprireAfterSeconds Integer 指定索引的过期时间
    storageEngine 针对document类型允许用户配置索引的额外存储引起

  • 可以通过getIndexes()或者getIndexSpecs()函数查看集合中的所有索引信息。

    db.COLLECTION_NAME.getIndexes()

  • mongodb不能修改索引，只能先删除再创建

  • 删除索引

    db.COLLECTION_NAME.dropIndex("索引名") // 删除单个索引
db.COLLECTION_NAME.dropIndexes() // 删除所有索引，_id字段的索引除外

  • 索引命中

    1. 如果创建的索引只包含1个字段，称为单字段索引。如果多个单字段索引的并集能覆盖查询语句需要的字段，则认为使用了交叉索引。
      1. 多key索引，当索引的字段为数组时。
    2. 如果创建的索引包含多个字段，称为复合索引。如果能命中复合索引，效率是最高的，命中复合索引要求回显的字段包含在复合索引的字段之中，因此这种场景也被称为复合索引覆盖。

# 重建索引

  • 索引是单独存储的，没有跟文档数据存在一起，在大量数据频繁变化之后会产生索引碎片，所以需要重建索引来整理。

  • 使用reIndex()函数重建索引，可以减少索引存储空间，减少索引碎片，优化查询效率，一般在数据大量变化后，会使用重建索引来提升性能。

    db.COLLECTION_NAME.reIndex()

# 使用索引的注意事项

  • 索引不是越多越好，因为它本身也会消耗存储空间，同时会增加插入、删除和修改操作的开销，另外数据库运行时也要消耗资源维护索引。

  • 当数据量达到万级才需要建索引，否则直接全集合扫描即可。

  • 尽量使用复合索引，而不是交叉索引。

  • 如果复合索引同时包含匹配条件和范围条件，那么匹配条件应放在范围条件之前，换句话说越精确匹配的越靠前。

    // 如果查询条件是这样的
db.test.find({username:"uuuu", age: {$gt: 30}})
// 那么应该这样建索引
db.test.createIndex({username: 1, age: -1})

  • 查询的时候尽可能只查询索引字段。

  • 对现有数据中的大表建立索引的时候，应采用后台运行的方式。

  • 由于索引是存储在内存中，所以应该确保索引的大小不超过内存限制，否是MongoDB会删除一些索引导致性能下降。

  • 查询限制（有索引但没法用），所以一定要用explain看下。

    • 正则表达式（最左匹配除外）及非操作符，如$nin，$not等
    • 算术运算符，如$mod等

  • 一个集合中的索引数量不能超过64个，索引名的长度不能超过128个字符，一个复合索引最多有31个字段

# 常用命令

  • 登录

    db.auth("username", "password")

  • 切换数据库

    use dbname

  • 查看当前集合中哪些字段建了索引

    db.collection_name.getIndexKeys()

# FAQ

  1. 即使没有唯一索引，还是会报写入冲突？

    原因：大概率是多个事务同时更新同一条记录导致，因为mongodb的事务是乐观锁，事务隔离级别为SI（Snapshot Isolation (opens new window)）。

    解决：官方建议在驱动层进行重试，如果没有则需要业务层重试。
